# Generic Modbus RTU UART Transport for any STM32 (CubeMX/Keil)

A drop-in Modbus RTU **slave + master** transport that runs on **any STM32
family** — F0, F1, F3, F4, F7, L4, G0, ... — using only:

- `HAL_UART_Transmit()` / `HAL_UART_Receive()` (blocking, byte-wise)
- `HAL_GetTick()` (the standard 1 ms HAL tick)

No DMA, no UART interrupts, no family-specific code. If your CubeMX
project has a UART and the HAL tick, this works.

## Why this exists

The protocol core of this repository (`src/modbus.c`, `src/modbus_diag.c`,
`src/modbus_master.c`) is **HAL-free plain C**. Proof: the CI native test
suite compiles those exact files with host `gcc` and runs 60+ protocol
tests against them (see `.github/workflows/build.yml`, `native-tests`
job). The only board-specific surface is:

1. `board_config.h` — a handful of integer defines (config header below)
2. four IO glue functions — shipped here as overridable `__weak` stubs
3. the UART transport — this example

## Files to copy into your CubeMX/Keil project

| From this repo | Purpose |
|---|---|
| `src/modbus.c` | RTU slave core + PDU dispatcher (FC 0x01–0x08, 0x0F/0x10, 0x14/0x15, 0x17, 0x2B) |
| `src/modbus_diag.c` | FC 0x08 Diagnostics sub-functions (optional but recommended) |
| `src/modbus_master.c` | Master PDU builders/parsers + transaction engine (optional, master only) |
| `inc/modbus.h`, `inc/modbus_diag.h`, `inc/modbus_master.h` | Core headers |
| `inc/digital_io.h`, `inc/analog_io.h`, `inc/relay.h` | IO glue prototypes (headers only) |
| `examples/generic_hal_transport/modbus_generic_uart_transport.c/.h` | This transport |
| `examples/generic_hal_transport/board_config.h` | **User config header — edit this** (family HAL include + Modbus map) |

You do **not** need `rs485.c`, `rs232.c`, `ethernet.c`, `modbus_tcp.c`,
`main.c`, or any other F407 board file.

Add the copied headers' directory to your include path and the `.c` files
to your build (Keil: add to a group; CubeIDE: they are picked up from
`Src`/`Inc` automatically).

## Wiring it up — slave

```c
#include "modbus.h"
#include "modbus_generic_uart_transport.h"

extern UART_HandleTypeDef huart3;   /* generated by CubeMX (any USART) */

static modbus_generic_uart_t modbus_port;

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_USART3_UART_Init();            /* 115200 8N1, no hw flow control */

    modbus_rtu_init(MODBUS_RTU_ADDRESS);              /* slave id = 1 */
    modbus_generic_uart_init(&modbus_port, &huart3, 115200);

    while (1) {
        modbus_generic_uart_slave_poll(&modbus_port); /* non-blocking */
        /* ... your application ... */
    }
}
```

## Wiring it up — master

```c
modbus_generic_uart_init(&modbus_port, &huart3, 115200);
modbus_master_init(&modbus_port.master_transport);

uint16_t regs[4];
uint8_t exc = 0;
modbus_status_t st =
    modbus_master_read_holding_registers(/*slave*/1, /*addr*/0, /*qty*/4,
                                         regs, &exc);

uint16_t echo = 0;
st = modbus_master_diag_query_data(1, 0x1234, &echo, &exc);   /* FC 0x08 */
```

Use one UART handle per role (slave **or** master) at a time; the
blocking byte-wise transport does not interleave both on one port.

## Bring-up test with a USB-TTL adapter (115200 8N1)

Connect adapter TX→MCU RX, RX→MCU TX, GND→GND. Send these hex frames
from any serial terminal (e.g. RealTerm, `pyserial`, Modbus poll tool)
and compare the replies byte-for-byte. CRCs below were computed with
CRC-16/Modbus (poly 0xA001, init 0xFFFF, low byte first) and verified
against the firmware's own `modbus_crc16`.

| # | Test | You send (hex) | Expected reply (hex) |
|---|------|----------------|----------------------|
| 1 | FC 0x08 sub 0x00 Return Query Data (loopback) | `01 08 00 00 12 34 ED 7C` | `01 08 00 00 12 34 ED 7C` (exact echo) |
| 2 | FC 0x03 Read Holding Register 0 (0x0000 after boot) | `01 03 00 00 00 01 84 0A` | `01 03 02 00 00 B8 44` |
| 3 | FC 0x06 Write Single Register 0 = 0x1234 | `01 06 00 00 12 34 84 BD` | `01 06 00 00 12 34 84 BD` (echo) |
| 4 | Re-read register 0 after test 3 | `01 03 00 00 00 01 84 0A` | `01 03 02 12 34 B5 33` |
| 5 | Exception: read register 0x0100 (out of range) | `01 03 01 00 00 01 85 F6` | `01 83 02 C0 F1` (FC\|0x80, exc 02 = Illegal Data Address) |
| 6 | Broadcast Clear Counters (FC 0x08 sub 0x0A) | `00 08 00 0A 00 00 C1 D8` | *(no reply — broadcast; counters cleared)* |

If frame 1 echoes correctly, framing + CRC + slave dispatch all work.
Frames 2–5 exercise the register map and the exception path; frame 6
verifies broadcast handling.

## The two Cortex-M7 (F7/H7) traps

If you port this to an **F7** (or any Cortex-M7) and later move away from
this blocking byte-wise transport, watch out for:

1. **D-cache maintenance with DMA.** Cortex-M7 has a data cache; the F4
   does not. If you switch RX/TX to DMA (CubeMX's default suggestion),
   the DMA engine reads/writes memory *behind the cache's back*. You must
   call `SCB_CleanDCache_by_Addr()` before TX DMA and
   `SCB_InvalidateDCache_by_Addr()` after RX DMA on cache-line-aligned
   (32-byte) buffers, or place the DMA buffers in a non-cacheable MPU
   region. The classic symptom of skipping this: *"the slave returns the
   previous response again"* — intermittent stale data. This example's
   byte-wise blocking approach sidesteps the whole category because the
   CPU touches every byte.

2. **Prompt `HAL_UART_Receive_IT` re-arm.** If you switch to
   interrupt-driven RX, `HAL_UART_Receive_IT()` must be re-armed
   immediately in `HAL_UART_RxCpltCallback()`. If re-arming is delayed
   (e.g. until after frame processing), a byte arriving in between causes
   an overrun (ORE) and reception silently stops. This example avoids the
   class by not using `HAL_UART_Receive_IT` at all.

## Notes & limits

- **Timing granularity:** T1.5/T3.5 use the 1 ms HAL tick rounded up
  (T3.5 = 2 ms at 115200). Fine for bring-up and moderate bus speeds.
  For strict compliance at high baud, upgrade to a µs timebase or the
  interrupt-driven engine from `src/rs485.c`.
- **8N1 only** in this example; adjust `huart->Init` in CubeMX for other
  formats and recompute your CRC test frames.
- The register map is plain memory unless you override the `__weak` IO
  stubs in `modbus_generic_uart_transport.c` with real drivers.
